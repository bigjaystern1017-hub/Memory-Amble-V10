Fixed Stage Layout with User-Controlled Pacing
Make changes to client/src/pages/amble.tsx only. Read ALL instructions before starting.
The core change: the lesson area should behave like a FIXED STAGE, not a scrolling feed. The stage has a set height that fills the available space between the header/progress bar and the input tray. Content inside the stage REPLACES — it never scrolls, never stacks, never pushes other content off screen.
THE STAGE LAYOUT
The visible screen should be divided into exactly these zones, top to bottom:

Top bar (pinned): App header with MemoryAmble logo, day title, sign in. Already exists — don't change.
Progress stepper (pinned): The Learn → Choose Place → Plant Images → Walk → Recall stepper. Already exists — don't change.
THE STAGE (fills remaining space): This is the main content area. It uses flex-grow or flex: 1 to fill all space between the progress stepper and the input tray. Inside this stage, content is vertically centered. NOTHING inside the stage scrolls. The stage contains at most:

One Timbuk speech bubble (the current/latest message)
One user reply bubble (if the user just responded)
One object card OR one recall card (when applicable)
That's it. Maximum 3 elements visible at once.


Input tray (pinned at bottom): The text input, confused button, mic button. Already exists — don't change positioning.

The key CSS: the stage container should be display: flex; flex-direction: column; justify-content: center; overflow: hidden; with a set min-height that ensures it fills the viewport. No scrolling inside the stage.
TRANSITIONS
When content changes in the stage (new Timbuk message, new object card, etc.), use a simple crossfade: old content fades out over 300ms, new content fades in over 300ms. No sliding, no pushing, no height changes. The stage container stays the same size throughout.
USER-CONTROLLED PACING
This is critical. After Timbuk's message finishes typing/revealing, if the beat requires user text input (placement, recall, stop naming), the input tray is already active — the user types and submits. That IS their advance action.
For beats that DON'T require text input (continue beats, transitions), show a button INSIDE the stage below Timbuk's message. The button label should match what the user is agreeing to do. NOT generic "Continue" — use the existing getContinueButtonLabel function which already returns contextual labels like "Yes, let us go!" or "I'm ready". The user taps the button, the stage crossfades to the next content.
REMOVE THE TRANSCRIPT DRAWER
Remove the "Review earlier messages" toggle and the transcript drawer entirely. Previous messages are gone — the user doesn't need them. This eliminates the scroll-within-scroll problem. If we need a transcript later, it can be a separate page, not inline.
OBJECT CARD BEHAVIOR
During placement beats: the object card appears INSIDE the stage, below Timbuk's current bubble. It should be compact — just the object name (bold), the stop name, and "Make it strange and vivid." in small text. No "Plant this image" header. No "Object 1 of 3" — the progress stepper and route panel already show progress.
During recall beats: same position in the stage, but with the amber/gold color scheme:

Card border: #D4A843
Card header text: #8B6914
Card background: #FFF8E7
"Walk to this stop. What's waiting there?" as the prompt text

After the user submits their answer for placement: the card briefly shows their response as confirmation (1.5 seconds), then fades. The route panel absorbs the placed item.
AUTO-FOCUS INPUT
After every Timbuk message that expects user text input, call .focus() on the input field after the message finishes its typewriter animation. The user should be able to start typing immediately without clicking into the input.
SOUND BANNER
The "Sound on" banner should automatically fade out after 4 seconds. Add: setTimeout that sets the banner's opacity to 0 with a CSS transition of 1 second, then removes it from DOM after the transition.
